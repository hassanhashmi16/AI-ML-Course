Do this: Produce one architecture diagram per project (boxes and arrows is enough) and a written paragraph per box explaining why that piece exists and why it was built that way.

0.2 Justify every dependency

For every library or framework choice you can't defend in one sentence ("I used X because Y, and the alternative Z would have cost me W"), that's a gap. Read the library's own "why/motivation" docs page until you can state the tradeoff, not just the name.

Do this: List every major dependency in both projects. Next to each, write the one-sentence justification. Anywhere you write "not sure" or "AI suggested it," spend 30–60 minutes closing that gap.

Weeks 2–5

Stage 1 — Foundations

The mechanical basics everything else sits on: how LLM APIs actually behave, how to control them, how to get structured output back out.

1.1 How LLM APIs actually work

An LLM API call is a stateless HTTP request: you send a list of messages (system/user/assistant roles), the model predicts the next tokens one at a time, and the response streams back token-by-token if you ask for streaming. There is no "memory" between calls — every previous message you want the model to see has to be resent in full each time. This is why conversation history grows your token bill linearly, and why "the model forgot" is almost always "the app didn't resend the context."

Go deeper: Claude prompt engineering docs Anthropic's interactive prompt engineering tutorial (free, 9 chapters + exercises)
Do this: Make three raw API calls (no framework) — one without system prompt, one with, one with streaming on — and diff the raw JSON responses so you see exactly what changes.
Self-check: Explain why a chatbot needs to resend the entire conversation on every message.

1.2 Tokens, context windows, and cost math

Models don't see characters or words — they see tokens, chunks produced by a tokenizer (roughly 4 characters per token in English). Cost and context-window limits are both counted in tokens, not characters, which is why a code-heavy prompt costs more than the same-length English paragraph (code tokenizes less efficiently). Every model has a maximum context window (input + output tokens combined); exceed it and the call fails or gets silently truncated depending on the SDK.

Go deeper: Claude docs — context windows section
Do this: Take a real chunk of your own text and a real chunk of code of similar character length; run both through a tokenizer and compare token counts. Calculate the $ cost of a 10-turn conversation at your model's per-token pricing.
Self-check: Why does a 100-message chatbot conversation get exponentially more expensive per turn, not linearly?

1.3 Prompt engineering patterns

Zero-shot (just ask), few-shot (show 2–5 examples of input→output), and chain-of-thought (ask the model to reason step by step before answering) are the three foundational patterns — everything else is a variation. System prompts set persistent behavior/role; XML-style tags (Claude-specific but widely imitated) help the model parse structure in long prompts unambiguously. The pattern that actually moves accuracy the most in practice is few-shot with examples that match your real edge cases, not generic ones.

Go deeper: DeepLearning.AI — ChatGPT Prompt Engineering for Developers (free, ~90 min, Andrew Ng + OpenAI's Isa Fulford) DeepLearning.AI — Building Systems with the ChatGPT API (multi-step prompting) Anthropic interactive tutorial
Do this: Take one real prompt from Project 1 (below) and run it three ways — zero-shot, few-shot with 3 examples, chain-of-thought — and log the output quality difference.
Self-check: When would few-shot beat chain-of-thought, and vice versa?

1.4 Structured output with Pydantic

Parsing free-text LLM output with regex is fragile — models phrase things inconsistently. The fix is to define a Pydantic schema (a typed Python class) and use the model's native structured-output / tool-calling mode to force the response into that shape, which the SDK then validates automatically. This is the difference between "the model said something like JSON" and "you have a guaranteed, type-checked object."

Go deeper: Pydantic official docs
Do this: Define a Pydantic model for a real use case (e.g., extract name/date/amount from an invoice-like text) and force the LLM to return exactly that schema. Break it on purpose (ambiguous input) and see how validation errors surface.
Self-check: What happens to your app if you don't validate LLM output and the model returns malformed JSON?

1.5 Async Python for I/O-bound API calls

LLM API calls spend almost all their time waiting on the network, not using your CPU — that's "I/O-bound" work, and Python's async/await lets your program do other things while waiting instead of blocking. asyncio.gather() fires multiple API calls concurrently so 5 calls that each take 2 seconds finish in ~2 seconds total, not 10. This matters the moment you have more than one LLM call per request (e.g., an agent calling multiple tools).

Go deeper: Real Python — Async IO in Python: A Complete Walkthrough FastAPI docs — Concurrency and async/await
Do this: Write two versions of a "summarize 5 documents" script — one sequential (for-loop with sync calls), one concurrent (asyncio.gather) — and time both.
Self-check: Why doesn't async speed up CPU-heavy work the same way it speeds up API calls?

1.6 Error handling & rate limits

Production LLM calls fail: rate limits (HTTP 429), timeouts, malformed responses. The standard fix is retry-with-exponential-backoff (wait 1s, then 2s, then 4s...) plus a max-retry cap, and treating your calls as idempotent where possible so a retry doesn't double-charge a user or double-send an email.

Go deeper: tenacity (Python retry library) — README/docs
Do this: Wrap your API calls with retry-with-backoff and deliberately trigger a rate limit (rapid-fire requests) to watch it recover gracefully instead of crashing.
Self-check: Why is naive immediate-retry (no backoff) actually dangerous under a rate limit?

🔨 Project 1

A CLI or small web tool (reuse your Next.js/FastAPI stack) that calls an LLM with structured, validated Pydantic output, streams responses, handles rate limits/errors with retry-backoff, and logs token usage/cost per call. Keep the UI minimal — the entire point is clean plumbing underneath.

Weeks 6–11

Stage 2 — RAG & Vector Databases

How to make an LLM answer from your own data instead of just its training knowledge.

2.1 What embeddings actually are

An embedding model turns text into a list of numbers (a vector, typically 384–3072 dimensions) such that semantically similar text ends up close together in that vector space. "Close" is measured with cosine similarity — the angle between two vectors, not their raw distance. This is why "dog" and "puppy" end up near each other even though they share no characters: the model learned to place them near each other during training based on how they're used in context.

Go deeper: Jay Alammar — The Illustrated Transformer (also covers embeddings intuition)
Do this: Embed 5 sentences (2 similar pairs, 1 unrelated) using any embedding API, compute cosine similarity between every pair, and confirm the similar pairs score higher.
Self-check: Why does cosine similarity ignore vector length (magnitude) and only care about direction?

2.2 Chunking strategies

You can't embed an entire document as one vector without losing detail, so you split it into chunks first. Fixed-size chunking (e.g., 500 tokens with 50-token overlap) is simplest but can cut a sentence or idea in half; recursive chunking splits along natural boundaries (paragraphs, then sentences) before falling back to fixed size; semantic chunking groups text by meaning shift. Bad chunking is the single most common silent cause of bad RAG answers — the right answer exists in the source document but never makes it into a retrievable chunk intact.

Go deeper: LangChain docs — Text Splitters concepts
Do this: Chunk the same document three ways (fixed-size, recursive, by-paragraph) and manually inspect 10 chunks from each — count how many cut a sentence or idea awkwardly in half.
Self-check: Why does overlap between chunks help, and what does too much overlap cost you?

2.3 Vector databases & indexing

Comparing a query vector against millions of stored vectors one-by-one (exact search) doesn't scale, so vector databases use approximate nearest neighbor (ANN) indexes like HNSW (a navigable graph structure) that trade a small amount of accuracy for massive speed. pgvector adds this capability directly to Postgres — meaning you don't need a separate database if you're already using Postgres, which fits your existing stack. Dedicated vector DBs (Qdrant, Pinecone, Weaviate) add features like built-in hybrid search and horizontal scaling that pgvector doesn't have out of the box.

Go deeper: pgvector official GitHub (README covers HNSW/IVFFlat indexing) DataCamp — pgvector tutorial Building a FastAPI + pgvector RAG backend (walkthrough)
Do this: Set up pgvector locally, insert 1,000+ embedded chunks, create an HNSW index, and compare query latency with vs. without the index.
Self-check: Why is an ANN index "approximate," and when would that inaccuracy actually matter?

2.4 Retrieval quality: top-k, hybrid search, reranking

"Top-k" retrieval just means grabbing the k most similar chunks — but similarity alone often surfaces chunks that are topically related without containing the actual answer. Hybrid search combines vector similarity with traditional keyword search (BM25) so exact terms (names, IDs, numbers) aren't lost to "semantic fuzziness." Reranking adds a second, more expensive pass — a cross-encoder model that looks at the query and each candidate chunk together (rather than independently, like embeddings do) and reorders by true relevance.

Go deeper: Ragas docs (covers retrieval metrics used to judge this)
Do this: Query your Stage 2 index with a question containing a specific number or name; check whether pure vector search misses it, then add BM25 hybrid search and compare.
Self-check: Why can a chunk be highly "similar" to a query but still not contain the answer?

2.5 The RAG pipeline end-to-end

Ingest (load raw docs) → chunk → embed each chunk → store vectors + metadata → at query time, embed the user's question → retrieve top-k similar chunks → stuff them into the prompt as context → generate the answer, ideally with citations back to source chunks. Every step is a place things silently break, which is exactly why Stage 3 (evals) exists — without measurement you can't tell which step is failing.

Go deeper: LangChain — RAG tutorial (official) DeepLearning.AI — Retrieval Augmented Generation course (free via Coursera)
Do this: Build the full pipeline once end-to-end with the simplest possible version of every step (fixed chunking, top-3 retrieval, no reranking) before optimizing any single piece.
Self-check: If an answer is wrong, how do you tell whether it's a retrieval failure or a generation failure?

2.6 Common RAG failure modes

"Lost in the middle": models pay less attention to information buried in the middle of a long context, so relevant chunks placed poorly in the prompt get ignored. Over-retrieval: stuffing in too many chunks dilutes signal and increases cost/latency for no accuracy gain. Stale embeddings: if source documents change but you don't re-embed, retrieval confidently returns outdated information.

Do this: Deliberately bury the correct chunk in the middle of a 10-chunk context vs. first position, and compare answer accuracy.
Self-check: Name two reasons a RAG answer can be wrong even when retrieval found the correct chunk.

A RAG knowledge assistant over a real corpus you actually care about (your own project docs, a domain you know, coursework) — FastAPI backend, Postgres + pgvector, simple frontend, with citations back to source chunks. You'll keep extending this project through Stages 3–6, so build it cleanly.

Weeks 12–15

Stage 3 — Evals

The stage that separates AI engineers from people who can call an API. This is your stated differentiation strategy — don't rush it.

3.1 Why evals exist

You can't unit-test a sentence the way you test a function's return value — the same question can have many correct phrasings, and LLM output is non-deterministic (same input, different output run to run). Evals solve this by scoring outputs against criteria (faithfulness to source, relevance to question) across a fixed test set, giving you a number you can track over time instead of a vibe. Without this, "I improved the prompt" is an opinion; with it, it's a measurement.

Self-check: Why can't you just eyeball 5 outputs and call the system "good"?

3.2 RAGAS core metrics

Faithfulness: does the generated answer only contain claims supported by the retrieved context (checks hallucination)? Answer relevancy: does the answer actually address the question asked, independent of correctness? Context precision: of the chunks retrieved, how many were actually relevant (are you retrieving noise)? Context recall: of the relevant information that exists, how much did retrieval actually find (are you missing things)? These four together let you localize a failure to either the retrieval half or the generation half of your pipeline.

Go deeper: Ragas — Quick Start (evaluating in 5 minutes) Ragas — Evaluate a simple RAG system (full tutorial)
Do this: Run all four RAGAS metrics against Project 2's outputs on 10 real questions and manually verify one low-scoring example by hand to confirm the metric caught a real problem.
Self-check: If faithfulness is high but answer relevancy is low, what does that tell you about where the bug is?

3.3 Building a golden eval dataset

A golden dataset is a fixed set of (question, ground-truth answer, expected source chunks) triples that represent real usage — not made-up trivia questions. Aim for 30–50 to start, covering easy cases, edge cases, and known-hard cases (ambiguous questions, questions with no answer in the corpus). You can bootstrap candidates with an LLM (synthetic question generation from your documents) but a human must verify every ground-truth answer, or you're just evaluating one model against another model's opinion.

Do this: Write 30 question/ground-truth pairs by hand for your Project 2 corpus, including at least 5 "trick" questions the system should refuse to answer because the info isn't there.
Self-check: Why is a 5-question eval set not enough to trust a score change?

3.4 LLM-as-judge pattern

Since grading "is this answer good" often can't be reduced to exact string matching, a common pattern uses a strong LLM as the grader, given the question, answer, and a rubric, to output a score. This scales evaluation but inherits the judge model's own biases (e.g., favoring longer or more confident-sounding answers) — so judge scores need periodic spot-checking against human judgment, especially early on.

Do this: Have an LLM judge grade 10 of your RAG answers on a 1–5 rubric, then grade the same 10 yourself, and compare where you disagree.
Self-check: Name one bias an LLM judge might have that a human grader wouldn't.

3.5 Tracing & observability

A trace records everything that happened during one request: the exact prompt sent, the retrieved chunks, tool calls made, tokens used, latency per step. Without tracing, debugging a bad answer means guessing; with it, you can see precisely which step (retrieval vs. generation vs. a tool call) produced the wrong input to the next step.

Go deeper: LangSmith documentation (tracing/observability)
Do this: Instrument Project 2 with LangSmith (or an equivalent), then intentionally ask a question you know it'll answer wrong, and use the trace to pinpoint the exact failing step.
Self-check: Without a trace, how would you have found that same bug — and how much longer would it take?

3.6 Regression gates / eval-driven development

A "ship gate" means: a versioned eval set, a numeric score, and an automated check that blocks a deploy if the score regresses below a threshold. This turns evals from a one-time report into a permanent safety net — the same role unit tests play in traditional software, adapted for non-deterministic systems. Teams that skip this ship by vibe, and vibe-shipped agents reliably regress within weeks as prompts/models/data drift.

Do this: Write a script that runs your RAGAS eval suite and exits with a failure code if any metric drops more than 5% from a saved baseline. Wire it into a simple CI check (GitHub Actions) on your repo.
Self-check: What's the difference between "I ran an eval once" and "I have a regression gate"?

🔨 Project 2, extended

Add a full eval harness to the RAG assistant: a versioned 30–50 question golden set, RAGAS scores tracked over time, and a CI check that flags regressions. This turns "I built a RAG demo" into "I built a RAG system I can prove works" — the exact artifact that differentiates you in the current hiring market.

Weeks 16–21

Stage 4 — Agents

Moving from a fixed pipeline to a system that decides its own next step.

4.1 What makes something an "agent"

A chain runs a fixed sequence of steps every time (A then B then C). An agent has a loop: it observes state, decides which action to take next (including whether to stop), takes the action, observes the result, and repeats — the number and order of steps isn't fixed in advance. This dynamic control flow is what lets an agent handle problems whose solution path can't be hard-coded.

Self-check: Give an example of a task better suited to a fixed chain than an agent, and explain why.

4.2 The ReAct pattern

ReAct (Reason + Act) interleaves explicit reasoning ("Thought: I need to look up X") with actions ("Action: call search tool") and observations ("Observation: search returned Y"), repeating until the model decides it has enough information to answer. Making the reasoning explicit in the trace (rather than hidden) is what makes agent behavior debuggable — you can literally read why it did what it did.

Do this: Manually trace through 3 ReAct-style thought/action/observation loops for a simple task (e.g., "what's the weather in the city where X was founded") by hand before writing code, so you internalize the loop shape.
Self-check: Why does making the "thought" step explicit (rather than hidden) help you debug agent failures?

4.3 Tool calling / function calling mechanics

You describe each tool to the model as a JSON schema (name, description, parameters) alongside your prompt. The model doesn't execute anything itself — it outputs a structured request ("call tool X with arguments Y"), your code actually runs the tool, and you feed the result back in as an observation for the next turn. Modern APIs support parallel tool calls (requesting multiple tools in one turn) to cut latency.

Do this: Give a model two tools (e.g., a calculator and a fake weather lookup) and a question that requires both; inspect the raw tool-call JSON the model produces before you execute anything.
Self-check: Who actually executes a tool call — the model, or your code?

4.4 LangGraph core concepts

LangGraph models an agent as a graph: a typed state object (a Pydantic model or TypedDict) flows through nodes (plain functions that read and update state) connected by edges (fixed or conditional — "if state.needs_more_info, go back to the retrieval node"). This graph structure, versus a plain while-loop, is what gives you visual clarity on complex control flow and lets you add cycles (loops) and branches cleanly.

Go deeper: LangChain Academy — Introduction to LangGraph (free course) LangGraph official GitHub (docs + examples) Real Python — LangGraph: Build Stateful AI Agents in Python
Do this: Build a 3-node LangGraph graph (e.g., retrieve → decide-if-enough-info → answer-or-loop-back) and draw its graph diagram before and after coding it to confirm they match.
Self-check: What's the difference between a fixed edge and a conditional edge?

4.5 Checkpointing & persistence

Long-running or multi-step agents need to survive a crash, a timeout, or a deliberate pause for human approval — checkpointing saves the graph's state at each step so execution can resume exactly where it left off, rather than restarting from scratch. This also enables human-in-the-loop patterns: the graph pauses at a node, waits for a human decision, then continues with that input folded into state.

Do this: Add checkpointing to your Stage 4 graph, deliberately kill the process mid-run, and confirm it resumes from the last checkpoint instead of restarting.
Self-check: Why does a customer-support agent handling a multi-day case need checkpointing, but a single-question chatbot doesn't?

4.6 Multi-agent orchestration with CrewAI

CrewAI structures multiple agents as a "crew," each with a role, goal, and backstory (a persona that shapes its behavior), plus tasks that get delegated between them. This beats a single agent when a problem naturally decomposes into specialist sub-roles (e.g., a "researcher" agent and a "writer" agent) where a single monolithic prompt would get confused trying to do both jobs at once — but it adds coordination overhead, so it's not the default choice for simple tasks.

Go deeper: DeepLearning.AI — Multi AI Agent Systems with crewAI (free course) CrewAI official learning platform CrewAI GitHub (docs + examples)
Do this: Build a 2-agent crew (researcher + writer) for a real task and compare its output quality/cost/latency against doing the same task with a single agent.
Self-check: When does splitting a task across multiple agents make things worse, not better?

4.7 MCP — Model Context Protocol

MCP is an open standard (created by Anthropic) that lets any tool/data source expose itself once as an "MCP server," and any compatible AI application can connect to it as an "MCP client" — instead of writing custom tool-integration code per framework per data source. It defines three primitives: tools (actions the model can call), resources (data the model can read), and prompts (reusable prompt templates). The payoff is decoupling: a tool you build once works across Claude, other MCP clients, and future frameworks without rewriting integration code.

Go deeper: Model Context Protocol — official docs & specification Anthropic — Introduction to Model Context Protocol (free course)
Do this: Build one minimal MCP server exposing a single tool (even something trivial like a local file search), connect it to a client, and call it end-to-end.
Self-check: What problem does MCP solve that plain function-calling tool schemas don't?

A multi-step agent (research assistant, support-triage bot, or similar) built in LangGraph with real tool use and checkpointing so it can resume after failure — not a single-shot chain wearing an "agent" label.

Weeks 22–25

Stage 5 — Agentic RAG + Trajectory Evals

Merge Stages 2–4 into your flagship project — the one you lead with in interviews and freelance pitches.

5.1 The agentic RAG pattern

Instead of a fixed retrieve-then-generate pipeline, the agent itself decides when to retrieve, what to search for, and whether one retrieval pass was enough — including rewriting the query if the first search came back weak, or doing multiple retrieval "hops" for questions that need information from several places. This handles the class of question plain RAG fails on: multi-part questions, or questions where the right search terms aren't obvious from the raw question text.

Do this: Find or construct a question your Stage 2 fixed-pipeline RAG gets wrong because a single retrieval pass wasn't enough, then fix it by giving the agent the ability to re-query.
Self-check: Give a concrete example of a question that needs two retrieval passes to answer correctly.

5.2 Trajectory evaluation

Grading only the final answer misses a whole class of bugs: an agent can get the right answer by calling the wrong tools, in the wrong order, or by getting lucky. Trajectory evaluation scores the path itself — did it call the tools it should have, in a sensible order, without redundant or wasted calls — which is what actually tells you whether the system will generalize to new questions or just happened to work on this one.

Do this: For 10 of your agent's runs, manually annotate the trajectory (tools called, in what order) against what the "ideal" trajectory would have been, and score the gap.
Self-check: Why can an agent produce a correct final answer via a trajectory you'd still call a bug?

5.3 Failure mode analysis

Not all failures are the same category: a retrieval miss (the right chunk was never found), a reasoning error (right information, wrong conclusion), or tool misuse (called the wrong tool, or the right tool with wrong arguments). Building a failure taxonomy from your own eval runs — tagging each failure by category — tells you where to actually spend your next week of work instead of guessing.

Do this: Run your eval set, tag every failing case by category (retrieval / reasoning / tool misuse), and count which category dominates — that's your next fix priority.
Self-check: Why is "the answer is wrong" not a useful bug report on its own?

Weeks 26–29

Stage 6 — Production & Deployment

Get the flagship project live, at a URL, surviving a stranger clicking around in it.

6.1 FastAPI production patterns

Dependency injection (FastAPI's Depends) lets you share things like DB connections and auth checks across endpoints without repeating code. Background tasks let you return a response to the user immediately while slow work (like an LLM call) finishes after the response is sent. Async endpoints (async def) matter specifically because LLM calls are I/O-bound — see Concept 1.5.

Go deeper: FastAPI docs — async/await & concurrency FastAPI docs — Dependencies
Do this: Refactor Project 4's endpoints to use dependency injection for the DB/auth, and move any slow non-response-critical work into a background task.
Self-check: Why would you use a background task instead of just making the endpoint slower?

6.2 Auth & rate limiting basics

At minimum: API keys or session tokens to identify callers, and per-user request quotas so one user (or a bug, or an attacker) can't exhaust your LLM budget or your database. This doesn't need to be enterprise-grade — it needs to exist at all, since an unprotected LLM endpoint is effectively an open invoice.

Do this: Add API-key auth and a simple per-key rate limit (e.g., 20 requests/hour) to Project 4, then test that exceeding the limit returns a clean error instead of crashing.
Self-check: What's the actual dollar risk of shipping an LLM endpoint with no rate limiting?

6.3 Docker fundamentals

An image is a frozen snapshot of your app plus everything it needs to run (Python version, dependencies, OS libraries); a container is a running instance of that image. A Dockerfile builds the image in layers, each layer cached, so "works on my machine" gets fixed by shipping the machine, not just the code. This is a light pass here — full depth (multi-stage builds, Compose) comes back in Phase 2, Stage 10.

Go deeper: freeCodeCamp — Docker & Kubernetes Fundamentals (free, hands-on)
Do this: Write a Dockerfile for Project 4, build it, run it locally in a container, and confirm it behaves identically to running it directly.
Self-check: What specifically does Docker fix that a requirements.txt file alone doesn't?

6.4 Deployment platform choice

Fly.io/Render/Railway are managed platforms — you push code, they handle servers, scaling, and TLS, trading some control for speed to a live URL. This is the right choice for a portfolio project; full Kubernetes control comes back in Phase 2 once the tradeoffs actually matter for a real production system.

Do this: Deploy Project 4 to one managed platform, end to end, and confirm it's reachable from a device that isn't yours.
Self-check: What are you giving up by choosing a managed platform over raw Kubernetes, and why is that an acceptable tradeoff right now?

6.5 Basic observability in production

Structured logging (JSON logs with consistent fields, not free-text print statements) makes logs queryable later. Track cost per request (tokens × price) and latency percentiles (p50/p95/p99 — the tail matters more than the average, since that's what angry users actually experience) from day one, even before you have real traffic, so you have a baseline.

Do this: Add structured logging to Project 4 capturing latency and token cost per request, and compute p50/p95 latency from a batch of test requests.
Self-check: Why does p95 latency matter more than average latency for user experience?
Phase 1 checkpoint (Stage 6 complete): Project 4 should be live at a real URL, with auth, rate limiting, logging, and cost tracking — something a stranger could use without you standing over their shoulder.

Weeks 33–38

Stage 8 — Math & Classical ML Foundations

Just enough linear algebra, probability, and classical ML to make Stage 9 click — not a full math degree.

8.0 The Python data stack: NumPy, pandas, Matplotlib, scikit-learn

These four are tools, not theory — you'll use them constantly from here through the rest of the curriculum, so get comfortable with them before the math gets heavy. NumPy gives you arrays and fast vector/matrix math (this is what "implement from scratch in NumPy" means in 8.5 — you're doing the linear algebra yourself, just not re-implementing array storage). pandas loads and cleans tabular data (CSVs, freelance usage logs, eval datasets) into DataFrames you can filter, group, and reshape — this is what you actually load Project 5's data with. Matplotlib is the plotting library behind every "plot this" instruction in this stage (8.1's vector transformation, 8.4's gradient descent steps, 8.6's decision boundary, 8.8's train/test gap) — without it those tasks have no visual output to inspect. scikit-learn gives you battle-tested implementations (regression, trees, forests, train/test splitting, metrics) to check your from-scratch code against, and to use directly once you've proven you understand the from-scratch version.

Go deeper: NumPy — official quickstart guide pandas — "10 minutes to pandas" (official) Matplotlib — pyplot tutorial (official) scikit-learn — Getting Started (official)
Do this: Load a real CSV (even a small export from your own freelance/portfolio data) into a pandas DataFrame, compute basic stats with pandas, plot one column with Matplotlib, and convert one column to a NumPy array for a manual calculation — touch all four in one script before moving to 8.1.
Self-check: Which of the four would you reach for to load a CSV, which for a from-scratch gradient calculation, and which to sanity-check your from-scratch model against a known-good implementation?

8.1 Vectors and matrices, geometrically

A vector is a point/arrow in space (you already used this in Stage 2 — an embedding is just a vector). A matrix multiply is a transformation of that space — rotating, scaling, or projecting vectors — which is exactly what happens at every layer of a neural network: each layer is a matrix multiply that reshapes the representation before the next layer sees it.

Go deeper: 3Blue1Brown — Essence of Linear Algebra (free video series, geometric intuition) Khan Academy — Linear Algebra (free, exercises included)
Do this: By hand, multiply a 2×2 matrix by a 2D vector and plot both the original and resulting vector — see the transformation visually.
Self-check: What does it mean, geometrically, for a matrix multiply to "rotate" a vector?

8.2 Dot product & cosine similarity, formalized

You used cosine similarity in Stage 2 as a black box; now derive it: the dot product of two vectors measures how much they point in the same direction, and dividing by the product of their magnitudes normalizes that into a -1 to 1 similarity score independent of vector length. This is the actual math underneath every RAG retrieval you've built.

Do this: Compute cosine similarity by hand (dot product, then divide by magnitudes) for two small 3D vectors, then verify your answer with code.
Self-check: Why does normalizing by magnitude matter — what would happen to search results if you skipped that step?

8.3 Probability basics: conditional probability & softmax

Conditional probability — P(A given B) — is the mathematical backbone of "predict the next token given everything before it," which is literally what an LLM does at every step. Softmax takes a list of raw scores (logits) and converts them into a valid probability distribution (all positive, summing to 1) by exponentiating and normalizing — this is the last step before a model "chooses" its next token.

Go deeper: Khan Academy — Probability library (free)
Do this: Take 5 raw logit values, compute softmax by hand, confirm they sum to 1, then verify with code.
Self-check: Why can't a model just pick the token with the highest raw logit directly, skipping softmax?

8.4 Gradients & why gradient descent works

A derivative tells you the slope (rate of change) of a function at a point; a gradient is the multi-dimensional version — the direction of steepest increase. Gradient descent repeatedly steps in the opposite direction of the gradient (steepest decrease) to find a minimum of a loss function — this is literally how every model you've used got trained, and it's the mechanical core you'll implement by hand in Stage 9.

Go deeper: Andrej Karpathy — Neural Networks: Zero to Hero (starts with exactly this)
Do this: Plot a simple parabola, pick a random starting point, and manually compute 3 gradient-descent steps toward the minimum using the derivative.
Self-check: Why does gradient descent move in the opposite direction of the gradient, not the same direction?

8.5 Linear regression from scratch

Linear regression fits a line (or hyperplane) to data by minimizing a loss function — typically mean squared error between predictions and actual values — using gradient descent to update the line's slope and intercept iteratively. This is the smallest possible complete example of "define a loss, compute gradients, update parameters," the pattern every model in this curriculum uses at larger scale.

Go deeper: Andrew Ng — Machine Learning Specialization (Coursera/DeepLearning.AI, free to audit)
Do this: Implement linear regression from scratch in NumPy (no scikit-learn) — loss function, gradient computation, update loop — on a toy dataset.
Self-check: What would happen to training if your learning rate (step size) were too large?

8.6 Classification & logistic regression

Logistic regression adapts linear regression for yes/no predictions by passing the linear output through a sigmoid function (squashes any number into 0–1, interpretable as a probability), and swaps mean-squared-error for cross-entropy loss (which penalizes confident-wrong predictions much more harshly). Cross-entropy loss is the same loss function used to train LLMs on next-token prediction — this is a direct bridge to Stage 9.

Do this: Implement logistic regression from scratch on a toy binary classification dataset (scikit-learn's built-in datasets work fine) and plot the decision boundary.
Self-check: Why is cross-entropy a better loss function for classification than mean squared error?

8.7 Trees & ensembles

A decision tree splits data on feature thresholds to make predictions and is easy to interpret but prone to overfitting on its own. Random forests train many trees on random subsets of data/features and average their predictions to reduce variance; gradient boosting trains trees sequentially, each correcting the previous ones' errors. These remain the default choice for structured/tabular data even in 2026, where deep learning usually isn't worth the added complexity.

Go deeper: scikit-learn docs — Ensemble methods
Do this: Train a single decision tree and a random forest on the same tabular dataset; compare accuracy and overfitting (train vs. test score gap).
Self-check: Why does averaging many overfit trees produce a model that doesn't overfit as badly?

8.8 Bias-variance tradeoff

High bias (underfitting): the model is too simple to capture real patterns, performing poorly on both train and test data. High variance (overfitting): the model memorizes training data noise, performing great on train but poorly on test/unseen data. Train/validation/test splits exist specifically to detect this gap — you tune on validation, and only check test once at the very end to get an honest read on real-world performance.

Do this: Deliberately overfit a model (e.g., a very deep decision tree with no depth limit) on a small dataset and plot train vs. test accuracy to see the gap widen.
Self-check: Why would tuning your model based on test-set performance invalidate your final evaluation?

Weeks 39–46

Stage 9 — Deep Learning & Transformers from First Principles

The math behind everything you built in Phase 1, learned once it has something real to anchor to.

9.1 The neuron / perceptron

A single artificial neuron computes a weighted sum of its inputs plus a bias, then passes that through a nonlinear activation function (like ReLU or sigmoid). The nonlinearity is not optional — stacking purely linear layers collapses mathematically into a single linear layer, no matter how many you stack, so nonlinearity is what actually lets a network learn complex, curved decision boundaries.

Go deeper: Karpathy — Neural Networks: Zero to Hero (micrograd lecture builds exactly this)
Do this: Implement a single neuron (weighted sum + ReLU) from scratch in plain Python, no libraries, and pass a few inputs through it by hand and in code to confirm they match.
Self-check: Why would a 10-layer network with no activation functions behave identically to a 1-layer network?

9.2 Backpropagation

Backprop is the chain rule from calculus applied systematically to a computation graph: to know how much a change in an early-layer weight affects the final loss, you multiply the local gradients along every step of the path from that weight to the output. Frameworks (PyTorch, etc.) do this automatically via autograd, but building a tiny one by hand once is what makes "the model learns by adjusting weights" stop being a black box.

Go deeper: Karpathy — "The spelled-out intro to neural networks and backpropagation: building micrograd" (free video + code)
Do this: Follow Karpathy's micrograd lecture and build a tiny autograd engine yourself (not just copy the code — retype it, break it, fix it) until you can explain every line.
Self-check: Why does backprop need to work backward from the output, rather than forward from the input?

9.3 Word embeddings before transformers

Word2vec learned static embeddings (one fixed vector per word, regardless of context) by training a shallow network to predict a word from its neighbors — establishing that "meaning" can be captured as position in vector space, the same idea underlying every embedding you used in Stage 2. The key limitation this motivates: static embeddings can't distinguish "bank" (river) from "bank" (money) — which is exactly the problem attention/transformers solve with contextual embeddings.

Do this: Read a summary of word2vec's training objective (predict-neighbor-words), then explain in your own words why a single fixed vector per word is a limitation.
Self-check: Why can't word2vec give "bank" two different meanings depending on sentence context?

9.4 Self-attention: Query, Key, Value

For each token, self-attention creates three vectors — Query (what am I looking for), Key (what do I offer), Value (what information do I actually carry) — and computes attention scores as the dot product of one token's Query against every other token's Key, then uses those scores to take a weighted average of all tokens' Values. This is the mechanism that lets "it" in a sentence dynamically figure out which earlier noun it refers to, based on the actual sentence content — not a fixed rule.

Go deeper: Jay Alammar — The Illustrated Transformer (the canonical explanation, with diagrams)
Do this: Work through Jay Alammar's Illustrated Transformer post with pen and paper, manually computing the Q/K/V calculation for a 3-word toy sentence.
Self-check: In your own words, what does the dot product between a Query and a Key actually measure?

9.5 Multi-head attention & positional encoding

A single attention head can only learn one "type" of relationship at a time; multi-head attention runs several attention computations in parallel with different learned Q/K/V weights, letting the model track multiple kinds of relationships simultaneously (e.g., grammatical role and topical relevance at once). Since attention itself has no built-in sense of word order (it treats input as a set, not a sequence), positional encoding injects position information into each token's embedding before attention runs.

Do this: Read the "multi-headed attention" section of the Illustrated Transformer and explain, without looking, why 8 heads see 8 different perspectives on the same sentence.
Self-check: If you removed positional encoding entirely, what would the model lose the ability to distinguish?

9.6 Transformer architecture end-to-end

A transformer block stacks: multi-head self-attention → residual connection (add the input back in, so gradients can flow through many layers without vanishing) → layer normalization (keeps activations stable) → a feed-forward network → another residual + norm. Encoder-decoder transformers (original translation-style) use both halves; decoder-only transformers (GPT-style, what modern LLMs use) use just the decoder stack, generating one token at a time.

Go deeper: Jay Alammar — The Illustrated GPT-2 (decoder-only architecture specifically)
Do this: Draw the full block diagram of a transformer layer from memory (attention, residual, norm, feed-forward, residual, norm), then check it against the Illustrated Transformer diagram.
Self-check: What problem do residual connections solve in a deep network?

9.7 From attention to GPT: causal masking & autoregression

Decoder-only models generate text autoregressively — predicting one token, appending it, then predicting the next based on everything so far. Causal masking prevents a token's attention from "seeing" future tokens during training (which would let the model cheat by looking at the answer), forcing it to learn genuine next-token prediction. This is literally the mechanism underneath every LLM API call you made back in Stage 1.

Go deeper: Karpathy — "Let's build GPT: from scratch, in code" (final lecture in the Zero to Hero series)
Do this: Follow Karpathy's "build GPT from scratch" lecture and train a tiny character-level GPT on a small text file on your own machine.
Self-check: Why would training without causal masking produce a model that looks great in training but fails at real generation?

9.8 Fine-tuning vs. prompting vs. RAG

Prompting (Stage 1) changes behavior with no weight updates — fastest, cheapest, but limited by context window and can't teach genuinely new knowledge patterns. RAG (Stage 2) injects external knowledge at query time without touching weights — good for facts that change often. Fine-tuning actually updates model weights on your own examples — worth it for changing style/format/behavior consistently, not for injecting facts (which RAG handles better and cheaper). Now that you've built all three in Phase 1, this decision framework has real weight behind it.

Do this: Take one problem from your Phase 1 projects and argue, in writing, which of the three approaches actually fits it best and why the other two would be worse choices.
Self-check: Why is fine-tuning usually the wrong tool for "the model doesn't know about our latest product docs"?

9.9 LoRA / PEFT

Full fine-tuning updates every weight in a model — expensive and requires storing a full new copy of the model per task. LoRA (Low-Rank Adaptation) freezes the original weights and injects small, trainable low-rank matrices alongside them, cutting trainable parameters by roughly 90% while achieving comparable results for most tasks. PEFT (the Hugging Face library) is the standard tool for applying LoRA in practice.

Go deeper: Hugging Face PEFT docs — LoRA conceptual guide Hugging Face — Smol Course: LoRA and PEFT (free)
Do this: Fine-tune a small open model with LoRA via Hugging Face PEFT on a toy dataset, and connect it explicitly back to a Phase 1 project — e.g., "here's the math behind why my chunking strategy in Project 4 worked."
Self-check: Why does freezing the original weights and only training small added matrices still work almost as well as full fine-tuning?

10.1 Docker deep dive

Beyond the basic Dockerfile from Stage 6: multi-stage builds separate the build environment (compilers, dev dependencies) from the final runtime image, producing smaller, more secure images. 10.2 Kubernetes basics

A Pod is the smallest deployable unit (one or more tightly-coupled containers). A Deployment manages a set of identical Pods, handling restarts and rolling updates. A Service gives a stable network address to a set of Pods even as individual Pods get replaced. Kubernetes exists to solve the problem Docker alone doesn't: keeping things running reliably when servers die, traffic spikes, or you need zero-downtime deploys.

10.3 CI/CD for ML

Continuous integration/deployment for ML extends normal software CI/CD (run tests on every PR, deploy on merge) with ML-specific checks: run your eval suite (Stage 3's regression gate) as part of the pipeline, not just unit tests, so a prompt change or model swap that hurts quality gets caught before it ships.

10.4 Model registries

A model registry (MLflow being the standard open-source choice) versions trained models the way git versions code — tracking which model version is in staging vs. production, its training data/hyperparameters, and its eval scores, so you can roll back to a known-good version if a new one underperforms.

10.5 Experiment tracking

Every training run has hyperparameters (learning rate, batch size) and resulting metrics (loss, accuracy) — experiment tracking tools (Weights & Biases, Comet, or MLflow's tracking component) log all of this automatically so you can compare 20 runs side by side instead of scrolling through terminal output trying to remember what changed between runs.

Stage 11 — Observability & Infrastructure

Know when production is breaking before your users tell you.

11.1 Monitoring philosophy: what to actually measure

Four categories matter for an AI system: latency (how long requests take), error rate (failed requests / total), cost (tokens × price, per request and aggregated), and eval-score drift (is quality silently degrading as data/models/prompts change over time). Most teams only monitor the first two; the eval-score drift category is what actually catches the "it works but got worse" failure mode unique to AI systems.

Do this: List, for Project 4, exactly which metric in each of the four categories you'd want an alert on, and why that specific threshold.
Self-check: Why can a system have 0% error rate and still be silently failing its users?

11.2 Prometheus & Grafana

Prometheus scrapes (pulls) metrics from your app on a schedule and stores them as time series; Grafana reads that data and renders it as dashboards, plus lets you define alerting rules (e.g., "fire an alert if p95 latency exceeds 2s for 5 minutes straight"). This is the standard open-source pairing for the "what to measure" list from 11.1.

Go deeper: Prometheus official docs — Overview Grafana official docs MLOps Zoomcamp — Module 5 covers monitoring with Prometheus/Grafana specifically
Do this: Instrument Project 4 to expose Prometheus metrics (latency, request count, error count), scrape them, and build one Grafana dashboard with an alert rule.
Self-check: What's the difference between Prometheus "scraping" metrics and an app "pushing" logs somewhere?

11.3 Feature stores

Training-serving skew happens when the feature computation logic used during model training doesn't exactly match the logic used at inference time in production — a subtle, common bug that silently degrades model quality. A feature store (Feast being the common open-source option) centralizes feature definitions so training and serving pull from the exact same computation, eliminating that class of bug by construction.

Do this: Identify one feature in a Phase 2 project that's currently computed differently (or in two different code paths) for training vs. inference, and explain the skew risk in writing.
Self-check: Give a concrete example of training-serving skew and how it would silently hurt predictions.

11.4 Infrastructure as code

Terraform lets you declare your infrastructure (servers, databases, networking) as version-controlled config files instead of manually clicking through a cloud console ("click-ops") — meaning your infrastructure is reproducible, reviewable in a pull request, and recoverable if something gets deleted by accident.

Go deeper: Terraform official documentation
Do this: Write a Terraform config that provisions the actual infrastructure Project 7 runs on (even a single server + database), destroy it, and recreate it from the config to confirm reproducibility.
Self-check: Why is "click-ops" infrastructure a liability even if it works fine today?

11.5 GitOps

GitOps (ArgoCD being the common tool) means your desired infrastructure/deployment state lives in a git repo, and a controller continuously reconciles the real system to match it — deploying happens by merging a commit, not by running a manual command, and drift (someone manually changing something in production) gets automatically detected and can be auto-corrected.

Do this: Set up ArgoCD (or read through its quickstart if a full setup is out of scope for your timeline) pointed at a git repo containing Project 7's deployment config, and manually drift the live state to watch it get flagged.
Self-check: Why does "deploy by merging a commit" reduce the risk of a bad manual deploy?
