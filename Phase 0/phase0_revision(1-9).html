1.Python Syntax Basics

Python is whitespace-delimited. No braces, no semicolons. Indentation is the block.

# This is a comment name = "Hassan" # no let, const, var — just assign age = 26 is_learning = True # Dynamic typing — same variable can change type (don't do this) x = 5 x = "hello" # allowed, but confusing # Multiple assignment a, b, c = 1, 2, 3 # Swap without temp variable a, b = b, a
Edge case: mixing tabs and spaces will crash your program. Pick spaces (PEP 8 says 4). Your editor should convert tabs automatically.

2.Control Flow

# if / elif / else score = 85 if score >= 90: grade = "A" elif score >= 80: grade = "B" else: grade = "C" # for loop — always over an iterable for i in range(5): # 0, 1, 2, 3, 4 print(i) # enumerate — when you need both index and value names = ["alice", "bob"] for idx, name in enumerate(names): print(f"{idx}: {name}") # zip — iterate two lists in parallel names = ["alice", "bob"] scores = [85, 92] for name, score in zip(names, scores): print(f"{name}: {score}") # while — use when you don't know the number of iterations x = 10 while x > 0: x -= 1 # List comprehension — the Pythonic way to transform/filter lists squares = [x**2 for x in range(10)] # [0, 1, 4, 9, 16, 25, 36, 49, 64, 81] evens = [x for x in range(10) if x % 2 == 0] # [0, 2, 4, 6, 8] pairs = [(a,b) for a in [1,2] for b in [3,4]] # nested = cartesian product
Why this matters later: comprehensions are the foundation for transforming data in pandas and PyTorch (tensor operations use the same mental model).

5.2 — __init__ is not a constructor (important)

When you write MyClass(), Python secretly runs two methods. Most people only know about the second one. Knowing both explains a common trap:

  • Step 1 — __new__ (the real constructor): creates a brand new, empty object in memory. Returns the object.
  • Step 2 — __init__ (the initializer): takes that empty object and fills it with your data (sets attributes).

You almost never override __new__. But the Singleton pattern does, and it reveals the trap dramatically:

# Singleton = a class that only ever creates ONE object. # No matter how many times you call Singleton() — same object every time. class Singleton: _instance = None # class variable — shared by all instances # __new__ is what CREATES the object. By overriding it, we can # intercept object creation and return an existing one instead. def __new__(cls, *args, **kwargs): if cls._instance is None: # first time? create it cls._instance = super().__new__(cls) return cls._instance # second time? return same old object # The trap: __init__ runs EVERY TIME you call Singleton(), # even when __new__ returned the same old object instead of creating a new one. def __init__(self, value): self.value = value # overwrites value on THE SAME object a = Singleton(1) # __new__: creates object. __init__: sets value=1 b = Singleton(2) # __new__: returns SAME object. __init__: sets value=2 (overwrites!) print(a is b) # True — they are literally the same object in memory print(a.value) # 2 — because b's __init__ overwrote a's value (same object) # The flow for b = Singleton(2): # 1. Python calls __new__(Singleton, 2) # 2. __new__ sees _instance already exists, returns it (no new object) # 3. Python then calls __init__(old_object, 2) # 4. __init__ sets self.value = 2 on the old object # 5. b points to the exact same object as a. # Result: a.value is now 2 even though we "created" a with value 1.
Trap: Singleton's __init__ overwrites your data every time you "make a new instance" — because it's always the same instance. Real singletons guard against this with a flag like _initialized = False and check it inside __init__.

5.3 — Inheritance

Child classes inherit (copy) everything from the parent. The parent's methods and attributes are all there. The child can then:

  • Override — replace a parent method entirely with its own version
  • Extend — call the parent's version first via super(), then add more
  • Add — define new methods the parent doesn't have

The code below shows all three patterns. The comments walk through what happens step by step.

# ── PARENT CLASS ── class Dataset: def __init__(self, path): self.path = path # parent sets up "path" def load(self): # This is a "fake abstract" trick: if someone creates a plain Dataset # and calls load(), it'll crash with NotImplementedError. # The child MUST override this with a real implementation. raise NotImplementedError("Subclasses must implement load()") def describe(self): return f"Dataset at {self.path}" # ── CHILD CLASS ── # The (Dataset) after ImageDataset means "ImageDataset inherits from Dataset." class ImageDataset(Dataset): # The child's __init__ has its OWN parameters. But it still needs the parent's # __init__ to run so that self.path gets created. That's what super() does. def __init__(self, path, transform=None): # super() = "get the parent class (Dataset) and call its version of __init__" # This sets self.path = path. WITHOUT THIS LINE, self.path would NOT exist. super().__init__(path) # Now the child adds its OWN attribute that the parent doesn't have. self.transform = transform # OVERRIDE: replace the parent's load() entirely. # The parent's version raised an error — this one actually works. def load(self): return "[image bytes]" # pretend we loaded real images # EXTEND: keep the parent's behavior AND add to it. # We call the parent's describe() first, then add more text. def describe(self): base = super().describe() # "Dataset at /data/images" return f"{base} with transform={self.transform}" # ── WHAT HAPPENS WHEN YOU USE IT ── ds = ImageDataset("/data/images", transform="resize") # ds.path came from the PARENT's __init__ (called via super().__init__(path)) print(ds.path) # /data/images # ds.transform came from the CHILD's __init__ (set directly) print(ds.transform) # resize # ds.load() uses the CHILD's version (overrides the parent's NotImplementedError) print(ds.load()) # [image bytes] # ds.describe() calls the PARENT's describe() via super(), then adds child's text print(ds.describe()) # Dataset at /data/images with transform=resize
Why super() matters: Without it, the parent's __init__ never runs and self.path never gets set. The child would crash with AttributeError: 'ImageDataset' object has no attribute 'path'. In Python 3, just write super() — no arguments needed. Never write super(ClassName, self) (that's Python 2).

5.4 — Multiple inheritance & MRO

Python supports multiple inheritance. The Method Resolution Order (MRO) determines which method is called. Use ClassName.__mro__ to see the order.

class A: def say(self): return "A" class B(A): def say(self): return "B" class C(A): def say(self): return "C" class D(B, C): pass d = D() print(d.say()) # "B" — B comes first in MRO print(D.__mro__) # D → B → C → A → object # If D explicitly calls super().say(), it follows the MRO: class D2(B, C): def say(self): return super().say() + " (from D2)" d2 = D2() print(d2.say()) # "B (from D2)" — follows MRO
Real-world MRO example: PyTorch's nn.Module uses MRO for hook chains. FastAPI's dependency injection uses MRO-like resolution. Know it exists, don't overuse multiple inheritance.

5.5 — @staticmethod vs @classmethod vs instance methods

Three kinds of methods, each with different access:

Method typeFirst paramCan access?
Instance methodselfInstance vars, class vars
@classmethodclsClass vars only (can't see self)
@staticmethodNothing specialNothing — just a function namespaced in the class
class ModelOps: version = "1.0" def __init__(self, name): self.name = name def instance_method(self): return f"model={self.name}, version={self.version}" @classmethod def from_pretrained(cls, model_id): """Factory — creates an instance using class context""" print(f"Loading {model_id}, lib version {cls.version}") return cls(model_id) # constructs instance @staticmethod def validate_lr(lr): """Stateless utility — no self, no cls""" return 0.0 < lr <= 1.0 # Usage m = ModelOps("bert") m.instance_method() # normal m2 = ModelOps.from_pretrained("bert-base") # factory — creates instance ModelOps.validate_lr(0.01) # True — no instance needed m.validate_lr(0.5) # also works (but calling on instance is weird)
When you'll see each:
— @classmethod: Pydantic's model_validate, PyTorch's torch.tensor factory methods
— @staticmethod: utility functions inside a class (like validation helpers)
— Instance methods: 90% of what you write

5.6 — Dunder methods (the ones that matter)

Magic methods let your objects behave like built-in types. These are the ones you'll actually use:

class Vector: def __init__(self, x, y): self.x = x self.y = y def __repr__(self): # what you see in the debugger / console return f"Vector({self.x}, {self.y})" def __add__(self, other): # enables v1 + v2 return Vector(self.x + other.x, self.y + other.y) def __eq__(self, other): # enables v1 == v2 return self.x == other.x and self.y == other.y def __len__(self): # enables len(v) return 2 def __getitem__(self, idx): # enables v[0] return (self.x, self.y)[idx] def __call__(self, scale): # enables v(2) — object is callable return Vector(self.x * scale, self.y * scale) v1 = Vector(1, 2) v2 = Vector(3, 4) print(v1) # Vector(1, 2) — __repr__ print(v1 + v2) # Vector(4, 6) — __add__ print(v1 == Vector(1, 2)) # True — __eq__ print(v1[0]) # 1 — __getitem__ print(v1(10)) # Vector(10, 20) — __call__
__call__ is everywhere: PyTorch models are callable objects: output = model(x). That's __call__ under the hood. Same with FastAPI route handlers.

5.7 — Properties (controlled attribute access)

Let you use getter/setter logic while keeping the dot-access syntax. You'll see these in Pydantic, dataclasses, and SQLAlchemy models.

class Temperature: def __init__(self, celsius): self._celsius = celsius # underscore = "private" by convention @property def celsius(self): return self._celsius @celsius.setter def celsius(self, value): if value < -273.15: raise ValueError("Below absolute zero") self._celsius = value @property def fahrenheit(self): return self._celsius * 9/5 + 32 # computed, no setter t = Temperature(25) print(t.celsius) # 25 — looks like attribute access print(t.fahrenheit) # 77.0 — computed property t.celsius = 30 # setter validates and sets # t.celsius = -300 # raises ValueError

5.8 — Encapsulation (it's just convention)

Python has no private keyword. Everything is public. Convention:

  • _name — "protected" (internal use, don't touch from outside)
  • __name — name mangling (Python renames it to _ClassName__name). Used to avoid collisions in inheritance, not for "privacy."
class Parent: def __init__(self): self.public = "anyone can see this" self._protected = "please don't touch" self.__mangled = "renamed to _Parent__mangled" class Child(Parent): def show(self): print(self.public) # works print(self._protected) # works (convention says don't) # print(self.__mangled) # AttributeError! print(self._Parent__mangled) # works but ugly — don't do this
Don't overthink this. Use _name for internal attributes. That's it. Libraries like Pydantic and FastAPI use _name extensively for internal state.

5.9 — Abstract Base Classes (ABC)

In 5.3 we faked "abstract methods" with raise NotImplementedError. ABCs are the real system. They do the same thing but enforce the rule at creation time instead of waiting until you call the method.

Think of an ABC as a contract: "you MUST implement these methods, or your class can't exist." This is everywhere in ML libraries — scikit-learn estimators all follow a base class contract, PyTorch's Dataset forces you to implement __len__ and __getitem__, and you'll likely write your own base classes for agent tools and model wrappers.

from abc import ABC, abstractmethod # ── ABSTRACT BASE CLASS ── # By inheriting from ABC, this class says "I'm a template, not something you use directly." class BaseTokenizer(ABC): # @abstractmethod means: any child class MUST implement this method. # The method body doesn't matter — it'll never run. 'pass' is fine. @abstractmethod def encode(self, text: str) -> list[int]: pass @abstractmethod def decode(self, tokens: list[int]) -> str: pass # ── CONCRETE CHILD ── # This class MUST implement both encode() and decode(). If it forgets one: # TypeError: Can't instantiate abstract class SimpleTokenizer with abstract methods encode class SimpleTokenizer(BaseTokenizer): def encode(self, text): # required by BaseTokenizer return [ord(c) for c in text] # convert chars to ASCII codes def decode(self, tokens): # required by BaseTokenizer return "".join(chr(t) for t in tokens) # ── THE KEY DIFFERENCE VS RAISE NotImplementedError ── # With ABC, the crash happens the MOMENT you try to create the object: # t = BaseTokenizer() # TypeError IMMEDIATELY — "can't instantiate abstract class" # With raise NotImplementedError (from 5.3), the crash only happens when you CALL the method: # ds = Dataset("/path") # works fine # ds.load() # crashes HERE instead t = SimpleTokenizer() print(t.encode("abc")) # [97, 98, 99] print(t.decode([97, 98, 99])) # "abc"
When to use ABC vs raise NotImplementedError: Use ABC when you're writing a framework that OTHER people will extend (they need the instant error). Use raise NotImplementedError for quick internal code where you just need a reminder to implement something later.

5.10 — Dataclasses (modern Python)

Writing classes that just hold data is tedious — you have to write __init__, __repr__, __eq__, and it's all boilerplate. @dataclass writes all of that for you automatically based on the type annotations you already wrote.

You'll use these constantly in AI engineering — training configs, model hyperparameters, API response containers, experiment tracking metadata. Pydantic (Step 11) is essentially dataclasses with validation on top.

# ── WHAT YOU WRITE WITH @dataclass ── from dataclasses import dataclass, field @dataclass # <-- this decorator does the magic class TrainingConfig: lr: float = 0.001 # type annotation + default value epochs: int = 10 batch_size: int = 32 device: str = "cuda" metrics: list = field(default_factory=list) # ← see note below # ── WHAT @dataclass GENERATES FOR YOU (you'd have to write this manually) ── # class TrainingConfig: # def __init__(self, lr=0.001, epochs=10, batch_size=32, device="cuda", metrics=None): # self.lr = lr # self.epochs = epochs # self.batch_size = batch_size # self.device = device # self.metrics = metrics if metrics is not None else [] # # def __repr__(self): # return f"TrainingConfig(lr={self.lr}, epochs={self.epochs}, ...)" # # def __eq__(self, other): # return (self.lr, self.epochs, ...) == (other.lr, other.epochs, ...) config = TrainingConfig(lr=0.01) # __init__ was auto-generated print(config) # __repr__ was auto-generated: TrainingConfig(lr=0.01, epochs=10, ...) # ── WHY field(default_factory=list) INSTEAD OF metrics: list = [] ── # This goes back to the edge case from Step 3: mutable default args. # If you wrote: # metrics: list = [] # Python creates ONE list at class definition time and ALL instances share it. # field(default_factory=list) calls list() every time a new instance is created, # giving each instance its OWN separate empty list. # This applies to any mutable type: list, dict, set. # ── FROZEN DATACLASS = immutable (like a tuple, but with named fields) ── @dataclass(frozen=True) class Point: x: float y: float p = Point(1, 2) # p.x = 3 # TypeError: cannot assign to field 'x' — frozen instance # Useful for immutable configs, hyperparameters that shouldn't change mid-training # ── SLOTS DATACLASS (Python 3.10+, more memory efficient) ── @dataclass(slots=True) # no __dict__, uses __slots__ internally class FastPoint: x: float y: float # Uses ~40% less memory — matters when you have millions of datapoints
When to reach for what:
— @dataclass: simple data containers, configs, internal DTOs (data transfer objects)
— frozen=True: when immutability matters (hashable, safe to use as dict keys)
— slots=True: when you have thousands/millions of instances
— Pydantic (Step 11): when you need validation, type coercion, serialization, or schema generation. Pydantic builds directly on dataclass concepts.

8.Files, JSON, try/except

# Context manager — auto-closes even if error with open("data.txt", "r") as f: content = f.read() # entire file as string lines = f.readlines() # list of lines # Writing with open("out.txt", "w") as f: f.write("hello\n") # JSON — you'll use this constantly (API responses, configs, datasets) import json data = {"name": "Hassan", "scores": [85, 92, 78]} # Write with open("data.json", "w") as f: json.dump(data, f, indent=2) # Read with open("data.json") as f: loaded = json.load(f) print(loaded["name"]) # "Hassan" # try/except — handle failures without crashing try: result = risky_operation() print(result) except FileNotFoundError: print("File not found") except ValueError as e: print(f"Bad value: {e}") except Exception as e: print(f"Unexpected: {e}") # catch-all — log it, don't silence it else: print("No error occurred") # runs only if no exception finally: cleanup() # always runs
JSON edge case: JSON keys must be strings. If you have json.load a dict with integer keys, they become strings. Use json.loads(data, object_hook=...) for custom handling.

9.Type Hints

Type hints don't affect runtime behavior — but they enable Pydantic validation (Step 11), autocomplete in your editor, and catch bugs before runtime with mypy.

from typing import List, Dict, Optional, Union, Tuple # Basic def greet(name: str) -> str: return f"Hello {name}" # Collections def process_scores(scores: List[float]) -> Dict[str, float]: return {"mean": sum(scores) / len(scores)} # Optional = value or None def find_user(id: int) -> Optional[dict]: if id in database: return database[id] return None # Union = one of several types def to_id(value: Union[int, str]) -> int: return int(value) # Type alias — avoid repeating complex types ConfigDict = Dict[str, Union[str, int, float]] def validate_config(cfg: ConfigDict) -> bool: return "lr" in cfg
Python 3.10+ shortcuts (which you should use):
— list[int] instead of List[int]
— dict[str, float] instead of Dict[str, float]
— int | str instead of Union[int, str]
— int | None instead of Optional[int]
In short: lowercase built-ins, pipe syntax for unions. This is the modern style.
